Optimización del Rendimiento (Objetivo: 100)
El rendimiento es el área más crítica. Las siguientes acciones están priorizadas por su alto impacto en la velocidad de carga y la experiencia del usuario.
1.1. Reducir el impacto del código de terceros (YouTube)
Problema: Los videos de YouTube incrustados directamente cargan más de 1.7 MB de JavaScript y CSS, bloqueando el hilo principal por más de 930 ms en móviles. Esto retrasa drásticamente el FCP y LCP, y aumenta el TBT.
Solución (Alta Prioridad): Implementar el patrón "Facade" (Fachada). En lugar de cargar el reproductor de YouTube completo, se carga una imagen en miniatura del video con un ícono de "Play". El video real solo se carga cuando el usuario hace clic en la miniatura.
Pasos de Implementación:
Reemplazar el <iframe> de YouTube con un <div> personalizado.
Este <div> debe contener la miniatura del video como imagen de fondo y un botón de reproducción.
Añadir un script que, al hacer clic en el <div>, lo reemplace dinámicamente por el <iframe> de YouTube con el parámetro autoplay=1.
Ejemplo de Código:
<!-- HTML para la fachada del video -->
<div class="youtube-facade" data-videoid="_LQbWkWlg6s" style="background-image:url('https://img.youtube.com/vi/_LQbWkWlg6s/hqdefault.jpg');">
  <div class="play-button"></div>
</div>

<!-- CSS para estilizar la fachada -->
<style>
  .youtube-facade {
    position: relative;
    cursor: pointer;
    background-size: cover;
    background-position: center;
  }
  .play-button {
    /* Estilos para el botón de play (puede ser un SVG) */
  }
</style>

<!-- JavaScript para cargar el iframe al hacer clic -->
<script>
  document.addEventListener('DOMContentLoaded', function() {
    const facades = document.querySelectorAll('.youtube-facade');
    facades.forEach(facade => {
      facade.addEventListener('click', function() {
        const iframe = document.createElement('iframe');
        iframe.setAttribute('src', `https://www.youtube.com/embed/${this.dataset.videoid}?autoplay=1`);
        iframe.setAttribute('frameborder', '0');
        iframe.setAttribute('allow', 'accelerometer; autoplay; clipboard-write; encrypted-media; gyroscope; picture-in-picture');
        iframe.setAttribute('allowfullscreen', '');
        this.parentNode.replaceChild(iframe, this);
      });
    });
  });
</script>
1.2. Mejorar la entrega de imágenes
Problema: Se están sirviendo imágenes mucho más grandes de lo necesario para las dimensiones de la pantalla del dispositivo, desperdiciando datos y retrasando el LCP. El ahorro potencial es de más de 323 KiB en móviles.
Solución: Utilizar imágenes responsivas con los atributos srcset y sizes para que el navegador descargue la versión más adecuada del archivo.
Pasos de Implementación:
Crear múltiples versiones de cada imagen (ej. pequeña, mediana, grande).
Modificar la etiqueta <img> para incluir srcset con las rutas a las diferentes versiones y sizes para informar al navegador sobre el tamaño de la imagen en diferentes anchos de pantalla.
Comprimir todas las imágenes utilizando formatos modernos como WebP (ya en uso, pero se puede mejorar la compresión) o AVIF.
Ejemplo de Código (para la imagen del proyecto):
<!-- Antes -->
<img src="/assets/2_1754167....webp" ...>

<!-- Después (Ejemplo conceptual) -->
<img 
  src="/assets/project-image-medium.webp"
  srcset="/assets/project-image-small.webp 480w,
          /assets/project-image-medium.webp 800w,
          /assets/project-image-large.webp 1200w"
  sizes="(max-width: 600px) 480px,
         (max-width: 900px) 800px,
         1200px"
  alt="Proyecto TRANSERVICA: Multicar Railway Convoy..."
  loading="lazy"
  class="...">
1.3. Eliminar recursos que bloquean el renderizado
Problema: El archivo CSS principal (/assets/index-D1hjCskL.css) bloquea el renderizado de la página, retrasando el FCP. Aunque es pequeño (19 KiB), su impacto es significativo en redes móviles lentas.
Solución: Extraer el CSS crítico (necesario para renderizar la parte visible de la página o "above the fold") e insertarlo directamente en el <head> del HTML. Cargar el resto del CSS de forma asíncrona.
Pasos de Implementación:
Utilizar una herramienta (como [Critical](https://github.com/addyosmani/critical)) para identificar y extraer el CSS crítico.
Insertar este CSS en una etiqueta <style> dentro del <head>.
Cargar la hoja de estilos completa de forma asíncrona.
Ejemplo de Código:
<head>
  ...
  <style>
    /* ... Contenido del CSS crítico aquí ... */
  </style>

  <!-- Carga asíncrona del CSS completo -->
  <link rel="preload" href="/assets/index-D1hjCskL.css" as="style" onload="this.onload=null;this.rel='stylesheet'">
  <noscript><link rel="stylesheet" href="/assets/index-D1hjCskL.css"></noscript>
  ...
</head>
1.4. Reducir el contenido JavaScript y CSS no utilizado
Problema: Se descargan grandes cantidades de código JS y CSS que no se ejecutan en la carga inicial. El ahorro potencial es de más de 1000 KiB en JS (principalmente de YouTube, solucionado con el punto 1.1) y 14 KiB en CSS.
Solución:
JavaScript: Aplicar "code splitting" a nivel de ruta en el empaquetador (Vite/Webpack). Esto significa que el código para secciones como "Proyectos", "Servicios", etc., solo se cargará cuando el usuario navegue a esas páginas o interactúe con ellas.
CSS: Utilizar herramientas como [PurgeCSS](https://purgecss.com/) durante el proceso de build para eliminar automáticamente todas las reglas de CSS que no se estén utilizando en los archivos HTML/JS.

1.5. Optimización de Fuentes Web (Fonts)
Problema: La carga de fuentes web puede causar un "flash" de texto invisible (FOIT) o un cambio de diseño (CLS) cuando la fuente finalmente se carga, como se observa en el reporte de escritorio (CLS de 0.018).
Solución:
Asegurar `font-display: swap`: Esto ya parece estar en uso, pero es crucial verificarlo. Garantiza que se muestre un texto de respaldo mientras se carga la fuente.
Preconectar al origen de las fuentes: Acelera la conexión inicial para descargar las fuentes más rápido.
Mitigar CLS con descriptores de fuente: Utilizar las propiedades CSS size-adjust, ascent-override, descent-override y line-gap-override para que la fuente de respaldo ocupe un espacio casi idéntico a la fuente web final.

Ejemplo de Código:
<!-- En el <head> -->
<link rel="preconnect" href="https://fonts.googleapis.com">
<link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>
<link href="https://fonts.googleapis.com/css2?family=Inter:wght@400;600;700&display=swap" rel="stylesheet">

<!-- En el CSS, para mitigar CLS -->
@font-face {
  font-family: 'Inter Fallback';
  src: local('Arial'); /* O la fuente de sistema que más se parezca */
  size-adjust: 103%;
  ascent-override: 90%;
  descent-override: 22%;
  line-gap-override: 10%;
}

body {
  font-family: 'Inter', 'Inter Fallback', sans-serif;
}

2. Optimización de la Accesibilidad (Objetivo: 100)
La accesibilidad es fundamental para que todos los usuarios puedan navegar el sitio. Las correcciones son relativamente sencillas pero de gran impacto.
2.1. Los botones no tienen nombres accesibles
Problema: Varios botones, probablemente los indicadores de carruseles y flechas de navegación, no tienen texto descriptivo. Un lector de pantalla los anunciaría como "botón", lo cual es inútil para el usuario.
Solución: Añadir el atributo aria-label a cada botón para describir su función de forma concisa.
Ejemplo de Código:
<!-- Antes -->
<button class="p-2 rounded-full ...">
  <!-- Icono SVG de flecha -->
</button>

<!-- Después -->
<button class="p-2 rounded-full ..." aria-label="Siguiente Proyecto">
  <!-- Icono SVG de flecha -->
</button>

<!-- Para los puntos del carrusel -->
<button class="w-3 h-3 rounded-full ..." aria-label="Ir al proyecto 3"></button>

2.2. Contraste de color insuficiente
Problema: El texto "QUOTE" en el botón verde y el texto "250 tons" sobre un fondo verde no tienen suficiente contraste, dificultando la lectura para usuarios con visión reducida.
Solución: Ajustar los colores para cumplir con el ratio de contraste mínimo de 4.5:1 (WCAG AA). Para el botón "QUOTE", se puede oscurecer el verde de fondo o usar un blanco más brillante. Para la etiqueta "250 tons", se debe oscurecer el fondo #3ea30f o cambiar el color del texto.
Se recomienda usar una herramienta como [WebAIM Contrast Checker](https://webaim.org/resources/contrastchecker/) para encontrar los colores adecuados.
2.3. Áreas táctiles demasiado pequeñas
Problema: Los botones de navegación de los carruseles (los puntos pequeños) son difíciles de presionar en dispositivos táctiles.
Solución: Asegurar que cada elemento táctil tenga un tamaño mínimo de 48x48 píxeles CSS. Esto se puede lograr aumentando el tamaño del propio botón o añadiendo padding para expandir el área clicable sin cambiar la apariencia visual.
Ejemplo de Código CSS:
.carousel-dot-button {
  width: 12px; /* w-3 */
  height: 12px; /* h-3 */
  /* Solución: Añadir padding para aumentar el área táctil sin cambiar el tamaño visual */
  padding: 18px; /* (12 + 18*2) = 48px */
  background-clip: content-box; /* Para que el padding sea transparente */
}

3. Mejora de las Mejores Prácticas (Objetivo: 100)
Esta sección aborda problemas de seguridad, confianza y experiencia de usuario que deben ser resueltos.
3.1. Asegurar que la Política de Seguridad de Contenido (CSP) sea efectiva
Problema: La CSP actual es insegura. El uso de 'unsafe-inline' y listas de hosts genéricas abre la puerta a ataques de Cross-Site Scripting (XSS).
Solución: Implementar una CSP estricta basada en nonces o hashes. Esto garantiza que solo los scripts autorizados por el servidor puedan ejecutarse.
Pasos de Implementación:
Generar un valor "nonce" aleatorio y único en el servidor para cada solicitud.
Añadir este nonce a todas las etiquetas <script> legítimas.
Configurar el encabezado CSP para permitir solo scripts con ese nonce.
Implementar Trusted Types para mitigar XSS basado en DOM.
Ejemplo de Encabezado CSP Estricto:
Content-Security-Policy: 
  script-src 'nonce-RANDOM_VALUE' 'strict-dynamic'; 
  object-src 'none'; 
  base-uri 'none';
  require-trusted-types-for 'script';

Nota: Implementar una CSP estricta es un proceso complejo que requiere una auditoría completa de todos los scripts en línea y de terceros. La solución de la fachada de YouTube (1.1) simplificará esto, ya que el script de YouTube se cargará dinámicamente y puede ser manejado por la política.
3.2. Errores de navegador y problemas en la consola
Problema: Se reportan errores en la consola, como violaciones de la política de permisos (accelerometer, gyroscope) y problemas con cookies de terceros.
Solución:
Permissions Policy: Añadir un encabezado Permissions-Policy en el servidor para deshabilitar explícitamente las APIs que no se utilizan. Esto mejora la seguridad y elimina las advertencias.
Cookies de terceros: El problema de las cookies de YouTube se resolverá con la implementación de la fachada (punto 1.1). A medida que los navegadores eliminan las cookies de terceros, depender de ellas es una mala práctica.

Ejemplo de Encabezado Permissions-Policy:
Permissions-Policy: accelerometer=(), gyroscope=(), microphone=(), camera=()

3.3. Imágenes con relación de aspecto incorrecta
Problema: El logo de la empresa se muestra con una relación de aspecto ligeramente distorsionada (1.33 mostrada vs 1.17 real).
Solución: Asegurarse de que las dimensiones (width y height) o las clases de CSS (como las de Tailwind) mantengan la proporción original de la imagen. En este caso, el uso de clases como h-24 y w-auto es correcto, pero la imagen original podría tener espacios en blanco. Se recomienda recortar la imagen original para que se ajuste a su contenido y luego dejar que el CSS la escale proporcionalmente.
